# 2026-06-12 stage-3-engine-built

BUILT + LIVE-VERIFIED: the three-head verify-select **file-editing engine** — the dragon's coding mode. `triad/coder.py` (`EditJob`, `read_repo`, `parse_edits`, `build_diff`, `apply_edits`) + `CommandOracle.check_workspace` (multi-file gate) + CLI `triad code "<task>" --repo --oracle --rounds --yes` + REPL `/code <task>`. **126/126 offline tests** (was 105; +21 coder tests). **LIVE on the free roster:** gave it a buggy `add(a,b)` (`a-b`) on a scratch repo gated by an oracle; Groq + OpenRouter both emitted whole-file edits, both PASSED the oracle in the sandbox (macos-seatbelt, network blocked), Groq selected, diff shown, applied → file became `a+b`. Generate→verify→select→apply, end to end, $0.

**Design pivot from [[stage 3 — fcc file-editing bridge]] (honest record):** v1 did NOT use git worktrees or the headless `claude` CLI / fcc. Instead — **in-memory whole-file edits**: each head returns the COMPLETE new contents of files it changes (`FILE: path` + fenced block); proposals are applied to in-memory copies of the repo file map and gated by running the oracle command against that map in the existing `sandbox.py`. Why better for v1: (a) weak free models are unreliable at tool-use loops + context-exact diffs but can emit a whole file in one shot — trivial to parse, nothing to mis-apply; (b) the real repo is NEVER touched until `apply_edits` on accept, so blast radius is zero without needing a worktree at all; (c) no fcc/`claude` binary dependency — uses the free roster directly. Safety: untrusted paths guarded (`_safe_relpath` rejects absolute + `..`), dotfiles skipped on read so `.env`/secrets never enter prompts, no-oracle → UNVERIFIED selection-only (never auto-applied), reuses the sandbox's honest tier reporting.

**Deferred (not needed for v1, still in the scope):** `fcc.py` + the headless-`claude`-via-fcc route — that's for complex MULTI-STEP agentic tasks the whole-file format can't express (large repos, many-file refactors). v1 covers single-shot bounded edits well.
